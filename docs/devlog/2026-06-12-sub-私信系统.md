## 数据模型（migration v11）

- `conversations(id, type, dm_key UNIQUE, created_by, created_at, last_message_id, last_message_at)`：`type` 为 'dm'/'group'（群聊预留，v1 只产出 dm）；`dm_key` 为 `'<小id>:<大id>'`，把"同一对用户唯一会话"降为 O(1) 唯一索引，INSERT 撞约束重查天然防并发；`last_message_*` 为反规范化的列表预览与排序字段。
- `conversation_participants(conversation_id, user_id, state, last_read_message_id, hidden_at, joined_at)`：请求态放参与者维度——请求本质是"会话在某人收件箱的位置"（发起方永远 inbox，接收方才可能是 request），群聊邀请将来也是参与者级语义。`hidden_at` 实现"删除会话/拒绝请求 = 只对自己隐藏"。
- `messages`：软删除墓碑同 posts 惯例（deleted=1、content 清空）；`message_media` 同 post_media 形态；`message_reactions` 主键 (message_id, user_id)——每人每消息一个回应，换 emoji 为 UPSERT 覆盖（X 行为）。
- 已读/未读/Seen/消息游标全部基于消息 id 而非时间戳：消息 id 单调递增，而模拟时钟可被设回过去。

## 后端（modules/messages 四件套）

- 13 个端点：find-or-create 会话、会话列表（inbox/requests 过滤 + 双段游标）、会话详情、消息列表（全项目唯一"倒序查升序展示"列表，`[id]` 单段游标）、发消息、标记已读、接受请求、删除会话、删除消息、加/撤回应、未读计数、SSE 流。
- 关键规则：屏蔽双向任一即 403（建会话与发消息都查）；非参与者一律 404 不泄露存在性；请求态在**首条消息发送时**判定（接收方未关注发送方 → 其参与行置 request），接收方回复 = 隐式接受；`markRead` 只增不减且钳制到会话最新消息；隐藏的会话在对方再发消息后自然重现（`last_message_at > hidden_at`）。
- 未读角标语义对照 X：`count` 为收件箱中含未读消息的**会话数**，`requestCount` 为待处理请求会话数，导航角标显示两者之和。
- DM 不写 notifications 表（X 也是独立未读体系）。
- 媒体互斥：`mediaRepo.attachedSet` 改为 `post_media UNION message_media`——一条媒体只能挂一处（帖子或消息），发帖与发消息两条路径自动互斥；消息媒体上限 4（对照 X）；软删消息保留 message_media 关联，墓碑仍占用媒体。

## SSE 实时推送（core/events/sse-hub.ts）

- SseHub 为 core 级基础设施：进程内连接表、25 秒注释行心跳、按用户推送（多标签页全收）、`closeAll()` 清场。事件四类：message:new / message:read / message:reaction / message:deleted，外加清场前的 shutdown。
- 认证：EventSource 带不了 Authorization 头，token 走 query 参数在路由内手动 `app.jwt.verify` + 核对 worldId。本机单用户应用，token 进访问日志的风险可接受。
- 热切换安全：WorldManager 新增 `onActivated(cb)` 钩子（新上下文就绪后触发，切到同世界的空操作不触发），index.ts 注册 `sseHub.closeAll()`——旧世界 token 重连即 401。进程 shutdown 同样清场。
- 路由 handler 用 `reply.hijack()` 自管原始连接，注销函数挂 `req.raw.on('close')`，规避 async handler 与流式响应的竞争。
- 发送者本人不经 SSE 回推（前端 mutation 已写穿缓存，回推会双插气泡）。

## 前端（features/messages/）

- 双栏布局：Layout 在 `/messages` 前缀启用宽模式（中栏 max-w-247 ≈ 988px、收起右栏），左栏 388px 会话列表（标题/新建按钮/收件箱-请求 Tab），右栏会话视图或空态。路由 `/messages` 与 `/messages/:conversationId` 同渲染 MessagesPage。
- 会话视图：消息倒序取、reverse 升序渲染、初始滚底；向上加载更早页时记录/恢复 scrollHeight 差防跳动；新消息到达且视口近底部时自动滚底；跨天插日期分隔条。请求态底部渲染接受/拒绝横幅替代输入框（接受前不上报已读，对方看不到 Seen，与 X 一致）；屏蔽态显示不可发送提示。
- 气泡：自己 `bg-x-blue` 白字右对齐、对方 `bg-x-input` 左对齐；悬浮出回应/删除操作；气泡下挂回应 emoji 行（自己的回应蓝边、点击撤销）；对方已读到的最后一条本人消息下显示"已读"；媒体复用 MediaGrid/灯箱。
- MessageComposer 新写（不复用发帖 Composer——@ 候选/链接卡/搜图/280 字限均为发帖语义）：自适应高度 textarea、媒体 ≤4、Enter 发送 / Shift+Enter 换行。
- 实时性：DmStreamProvider 在 Layout 挂一条 EventSource（以 user.id 为依赖，切账号/登出重建）；message:new 写穿 `['dm-messages']` 缓存第一页并失效会话列表/角标，其余事件低频直接失效；断线回退 10 秒轮询，角标与会话列表保持 30 秒轮询兜底。虚拟用户（第二阶段）不依赖 SSE，纯轮询可完成一切操作。

## 遇到并解决的问题

- PowerShell 反引号是转义符，行内 node -e 的 SQL 模板字符串被吃掉——查库脚本改写成临时 .cjs 文件执行。
- `/api/users/:handle/posts` 的 items 是 PostView 而非 TimelineItem，清理脚本误取 `$it.post.id` 导致删除 404，改 `$it.id` 后清理成功。
- `useI18n().t` 的 key 为字面量联合类型，函数参数里传 `string` 类型的 key 不过类型检查——调用点直接写字面量。
