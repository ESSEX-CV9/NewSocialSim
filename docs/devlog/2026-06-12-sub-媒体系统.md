# 开发日志 2026-06-12 · 媒体系统四期（A 图片 / B 视频 / C URL 引入 / D 关键字搜图）

媒体系统经与用户充分讨论后定为四期推进：A 图片地基 → B 视频 → C URL 引入+OG 链接卡片 → D 关键字搜图（Pinterest/Pixiv 等 7 源，源自用户提供的 image_searcher 适配器，服务于编造虚拟帖与未来 AI 虚拟用户 tool call 取图）。关键决定：外部媒体一律下载入库（世界自包含）；搜图引导登录不用 Playwright 不写注册表，改用本机浏览器 + CDP；Pinterest 匿名搜索为主。四期全部交付，涉及 migration v7–v8。

### 后端

- migration v7：`media` 表（owner/type/file_name/mime/宽高/大小/source/origin_url）+ `post_media(post_id, media_id, position)` + users 加 `avatar_media_id`/`banner_media_id`。文件存 `data/worlds/<id>/media/<mediaId>.<ext>`，路径每次经活动世界现算（热切换安全）。
- 新 media 模块四件套：`POST /api/media/upload`（@fastify/multipart，png/jpeg/webp/gif 白名单，≤10MB，image-size 探测宽高，探测失败视为非法图片）；`GET /api/media/:id/file?w=`（公开流式 + immutable 缓存头；`?w=` 校验防跨世界缓存污染，不一致 404）。入库用两步法：INSERT 占位取 id → 回填 `<id>.<ext>` 文件名 → 写盘，写盘失败回滚删行。
- 发帖支持 `mediaIds`（本人所有、一条媒体只能挂一个帖子；头像/Banner 不占名额）；有媒体时允许空文案；软删帖保留媒体与关联。`PostView.media` 在 buildViews 唯一组装点批量注入，引用嵌入帖同样生效、墓碑帖给空数组。
- 新增 `GET /api/users/:handle/media`（含回复的带媒体帖，游标分页，媒体 Tab 数据源）。
- 头像/Banner：`PATCH /api/users/me` 接受 `avatarMediaId`/`bannerMediaId`（null 恢复默认）；`UserSummary` 加 `avatarUrl`，全部组装点同步；`UserProfile` 另带 `bannerUrl` 与两个 media id。
- 新依赖：`@fastify/multipart`、`image-size`（纯 JS）。

### 前端

- http 封装支持 FormData（不设 Content-Type 不 stringify）。
- Composer：图片按钮选图即传、缩略图条可删、有图允许空文案、上传中禁发。
- 新组件 MediaGrid（X 式宫格：1 张按原比例限高、2 张两列、3 张左一右二、4 张 2×2）与 MediaLightbox（黑底大图查看器，多图左右切换、Esc 关闭）；PostCard 与引用嵌入卡均渲染。
- Avatar 组件支持图片模式（有 avatarUrl 渲染 img，否则保持哈希色块），全站调用点接入；本地账号快照（localStorage）自动携带新字段。
- 个人主页：Banner 有图显示图片；媒体 Tab 为三列缩略图网格（分页加载）；编辑资料表单新增头像/横幅上传与恢复默认。

### 遇到并解决的问题

- 用户验收发现帖子图片全裂、发帖框预览不显示：`GET /api/media/:id/file` 返回 200 但 body 为 0 字节。根因是 Fastify 5 的 async handler 竞争——handler 调用 `reply.send(stream)` 后未 `return reply`，async 函数 resolve 出的 undefined 被当作响应载荷、抢在流管道前以空 body 结束响应。JSON 接口同步序列化不受影响，**流式响应必须 `return reply.send(stream)`**。隔离复现确证后一行修复。验证教训：流式接口的断言必须包含 body 字节数，只看状态码与响应头会漏掉此类问题。
- 修复后浏览器仍可能显示裂图：空响应已按 `immutable` 缓存头被缓存，需 Ctrl+F5 强刷一次。

### 提交记录

- 上传支持 mp4/webm，上限 100MB（multipart 全局限额同步上调；图片的 10MB 限制在 service 层单独校验）。视频走流式落盘不进内存：先写 `pending-<pid>-<计数>` 临时文件，入库取 id 后改名为 `<id>.<ext>`；multipart 限额触发时流被静默截断，落盘后显式检查 `truncated` 与字节数。宽高存 NULL（不引入视频探测依赖）。
- 文件端点支持 HTTP Range：`bytes=a-b` 与 `bytes=-N` 后缀形式均解析，返回 206 + `Content-Range`/`Accept-Ranges`，不合法的 Range 按整文件 200 响应；视频拖进度条依赖此能力。
- 前端：Composer 文件选择 accept 扩展至视频；预览条与帖子内（MediaGrid）均渲染内联 `<video controls preload="metadata">`，点击不冒泡到整卡跳转；个人主页媒体 Tab 视频缩略图加播放角标。
- 验证（流式断言含 body 字节数）：5000 字节视频上传后整文件 GET 返回 5000 字节；`Range: bytes=0-99` 返回 206 + `content-range: bytes 0-99/5000` + 恰 100 字节；`bytes=-50` 返回 50 字节。

### URL 引入媒体

- 新增基础设施 `core/safe-fetch.ts`：`assertPublicHttpUrl`（仅 http/https，拒绝 localhost/.local/私网与链路本地 IP 段/IPv6 环回与 ULA，重定向后对落点二次校验）+ `fetchWithLimit`（AbortSignal 超时 + 读流字节上限，超限即断；fetch 自身的网络错误与超时统一映射为业务 400）。
- `POST /api/media/from-url { url, source? }`：下载（10MB）→ content-type 白名单，不可信时用 image-size 嗅探真实格式兜底 → 复用 `createFromBuffer` 入库。服务端维护防盗链 Referer 映射（i.pximg.net/s.pximg.net → pixiv），D 期搜图选图直接复用此入口。
- Composer 新增"通过链接添加图片"（ri-link 按钮 + 行内输入框），与上传图共享限额规则。

### OG 链接卡片（migration v8）

- `link_cards` 表按 URL 缓存元数据，失败也缓存（status='failed'）防止反复抓取。
- 新 link-cards 模块（service+repo 两件套，无路由）：发帖时提取正文首个 URL（去尾随标点），抓 HTML → 正则提取 og:title/description/image/site_name（容忍属性顺序颠倒与单双引号，HTML 实体反转义，charset 按响应头/meta 自适应解码）→ `<title>` 兜底 → og:image 相对路径解析为绝对后下载入库（source='linkcard'）。`resolve` 全程吞错——链接卡片失败不阻断发帖。
- `posts.create` 改 async：发帖前预抓卡片（有媒体时跳过）；`buildViews` 批量查缓存注入 `PostView.linkCard`（有媒体或墓碑帖为 null，与 X 行为一致）。
- 前端：正文 URL 渲染为外链（去协议截短显示、stopPropagation 新标签打开）；新组件 LinkCard（X 式上图下文卡片：站点名/标题单行/描述两行截断，点击打开外链）。

### 遇到并解决的问题

- `fetch` 自身抛出的网络错误（DNS 失败/超时的 TypeError/AbortError）未被映射为业务错误，from-url 对不可达 URL 返回 500；在 safe-fetch 中包装为 ValidationError 后统一 400。
- PowerShell 5.1 不支持三元运算符（`? :` 为 PS7 语法），验证脚本改用 if/else。
- SSRF 验证：127.0.0.1/192.168.x/localhost/file:///[::1] 五种形态全部 400。

### media-search 模块

- 七个搜图适配器从 image_searcher/ 的 CommonJS 原型移植为 TS（统一 SearchAdapter 接口与候选格式）：pinterest（匿名 BaseSearchResource 优先、`x-pinterest-pws-handler` 绕过头保留、可选 Cookie 走认证版；主图取 736x 而非 orig——原图经代理常超时且可能超 10MB 入库上限）、pixiv（底层库由停更的 pixiv-api-client 换为活跃的 @book000/pixivts，refresh token 认证 + 50 分钟重刷；popular_desc → popular-preview → N users入り 阈值递减 → date_desc 四层降级保留；R-18/R-18G 标签过滤）、danbooru/gelbooru/yandere（标签解析基建 TagResolver 共用；gelbooru API 已强制要求 api_key+user_id）、pexels（需免费 key）、wikimedia。
- 路由：`GET /api/media-search?q=&source=&rating=`（单源直查/全可用源 Promise.allSettled 并行，单源失败静默；rating 三态 safe/all/r18，缺省取世界设定）、`/sources` 状态（含 `supportsRating` 能力声明）、`/preview?url=`（公开，仅白名单 host：i.pximg.net/s.pximg.net 带 Referer 转发）、`GET/PATCH /config`（凭证打码返回，PATCH 局部合并）。搜图候选经 C 期的 from-url 入库挂帖，AI 虚拟用户"一次检索拿图发帖"= search → from-url → createPost 三个 API。
- 实例配置 `data/media-search.json`（读容忍 BOM、写无 BOM）。`proxy` 字段经 undici 全局 ProxyAgent 作用于本进程全部出站 fetch（含 pixivts 内部）；取消代理需重启服务。
- **Pixiv CDP 引导登录**（不引入 Playwright、不写注册表）：服务端生成 PKCE → 探测本机 Chrome/Edge → spawn（独立调试端口 + 临时 profile）打开登录页 → 原生 WebSocket 连 DevTools 协议监听各页面导航 → 捕获 `pixiv://account/login?code=` 自动换 refresh token 写入配置 → 清理进程与临时目录（EBUSY 重试）。5 分钟总超时；手动粘贴 code/完整回调 URL 兜底；浏览器探测失败时返回 loginUrl 供手动打开。
- 内容分级：`WorldMeta.contentRating`（safe/all）由创建世界表单选择，旧世界缺省 safe，作为单次搜索 rating 的默认值；booru 系映射 rating 标签（safe 排除 / r18 仅 `rating:e|explicit`），pixiv 按 R-18 标签三态过滤；R-18G 由实例配置 `pixiv.allowR18G` 单独把门（设置页复选框即点即存）。

### 遇到并解决的问题

- @book000/pixivts 是 CJS 包且 ESM 命名导出静态探测失败（`__exportStar` 形态），运行时报 "does not provide an export named 'Pixiv'"；类型用 `import type`、运行时经 `createRequire` 加载解决。
- Node fetch 不走系统代理，七源在大陆网络直连全部 0 结果；经配置接口写入 `proxy: http://127.0.0.1:7897` 后 pinterest（16 条）/wikimedia/yandere 全通。danbooru 对代理出口 IP 弹 Cloudflare 人机挑战（网络相关，代码无解）；gelbooru 匿名 API 返回 401（已改为需凭证）。
- Pinterest orig 原图经代理 15 秒超时（且可能超 10MB），736x 变体 1 秒即回：适配器主图改取 736x，from-url 超时放宽到 30 秒。
- 编辑器/输出层把正则里的 BOM 转义还原成真实 BOM 字符的反复事故，最终用 PowerShell 按字符码拼接写入源码。
