# 开发日志 2026-06-18 · 编辑器 UX 与布局持久化

内容池面板（见 `2026-06-18-内容池面板.md`）之后的一轮编辑器打磨：时间轴交互、检视器原帖、窗格外观与窗口比例，以及一段反复返工五轮才收敛的**布局持久化**。提交 `65f13e4`（主体）+ 后续两个 UI 小修。

## 时间轴

- **有界窗口模型**：顶部改为 `左偏移 ｜ 中轴时间 ｜ 右偏移 + 单位下拉（分/时/天，默认 12/12 时）`。可见范围 = `中轴 − 左` 到 `中轴 + 右`，横轴/滚动条恰好覆盖这段，滚到头即窗口边缘。改边界/单位/中轴时自动把缩放调到"整窗铺满视口"，再 Ctrl+滚轮放大看细节。中轴仅在点「回到现在」后跟随流速前进，否则冻结。替换掉旧的 7 天稳定轴 + 无限滚动 + 后台预取。
- **缩放锚定**：Ctrl+滚轮缩放锚到光标位置、缩放滑块锚到视口中心，缩放点不再漂移。
- **刻度自喂养无限右滚修复**：旧实现的刻度线按滚动位置（`gxMin/gxMax`）生成绝对定位满高 div，滚到右缘会画出超过 `trackWidth` 的刻度，撑大 `scrollWidth` → 抬高 `gxMax` → 再画更靠右的刻度，自我喂养、无限右扩（与时钟无关，暂停照旧）。改为刻度/块/播放头一律只在 `[windowFrom, windowTo]` 窗口内渲染。
- **自定义横向滚动条**：原生拉手尺寸 ∝ 视口/内容，对缩放呈双曲，高缩放下缩成一个小点、难用。改为隐藏原生横向条（`overflow-x:hidden`，纵向仍用原生），在时间轴下方自绘一条，拉手宽度 `max(44px, 视口/内容比例)`——永不缩成点，可拖、点轨道跳转；Alt+滚轮横滚照旧。横向滚动全部走 `el.scrollLeft` 程序化设置（`overflow-x:hidden` 下仍生效）。
- **轨道分割线右侧补齐**：轨道宽取 `max(窗口像素宽, 视口宽)`，右侧不再断。

## 布局持久化（五轮返工的根因链）

布局存档反复丢失，连改五轮才收敛。每轮修一个真因、又暴露下一个，记录于此以免重蹈：

1. **整体覆盖写**：前端持有完整 doc、每次写整体 PUT 覆盖。`docRef` 未加载好时一次写就把磁盘其它字段冲空，且"读空→写空"粘连死循环。→ 后端改**按字段合并写**：`PUT/DELETE /api/layouts/saved`、`PUT /api/layouts/preset`、`PUT /api/layouts/last`，后端 read-modify-write 单字段，前端不再整体覆盖。修好 #3（命名布局丢失）。
2. **预设/存档/last 三套割裂**：只有"预设"上下文才捕获修改，改命名存档切走再回来丢。→ 前端统一**「活动布局（预设 | 命名存档）」**判别式模型：修改回写到活动布局自身的槽，切走定格、切回取它存的版本；命名存档随编辑自动更新（与用户确认是期望行为）。
3. **水合门闩卡死**：所有"捕获/回写"受 `hydratedRef` 门闩，而它只在 `loadLayouts` 成功后由一个 `setTimeout` 翻 true；`loadLayouts` 在应用布局时若抛异常（`fromJSON` 半坏后 `applyPreset` 再抛），那行翻门闩永不执行 → 门闩永久 false、捕获全程静默空转 → 切回必是原始布局。证据：磁盘 `presets:{}`、`lastActive:undefined`（受门闩的写从没落盘），而不受门闩的 `saveCurrent` 能写。→ `loadLayouts` 把应用布局包 try/catch、**GET 成功即同步翻门闩**（与 apply 解耦，必翻）；加载改 `apiReady` 触发（后端自解析活动世界，不等 `world.id`）；`attempt()` 包 try/catch。修好 #2。
4. **并发写竞态冲空 saved**：修好 #2 后，每次拖拽/切换并发发多个 PUT，各自对整文件 read-modify-write 且无串行；非原子 `writeFile` 写到一半被并发 `readLayouts` 读到半截 → 解析失败 → catch 返回空 doc → 把 `saved:[]` 落盘。这正是"修好 #2 反而回归 #3"的机制（#2 修好前捕获被门闩卡死、几乎不写盘，故无并发）。→ 后端 `writeLayouts` **原子写**（写 `.tmp` 再 `rename`）；`readLayouts` 区分 ENOENT（返空 doc）与已有文件解析失败（**抛错、中止本次写、不冲数据**）；加模块级 `lockLayouts` 互斥串行化四个端点的 RMW。并发冲压（40×preset + 40×last + 10×saved 同时发）验证：12 条命名布局全在、预设全写入、无 lost update。

最终架构：后端是布局存档的权威源、按字段原子合并写 + 串行 RMW；前端只发"这一条字段"、持本地缓存供读、切换时本地缺失从后端自愈。配套：加载失败不开闸不覆盖、定时重试；底部状态条显示「当前布局」；「恢复默认」重置当前预设；重开恢复 `last` 与 `lastActive`（旧 `lastPresetId` 自动迁移）。
