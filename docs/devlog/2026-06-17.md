## 施工路线图与原则

四步阶梯过粗，展开为单提交级原子步施工文档 `docs/m5-x-roadmap.md`：每步含目标 / 改动 / 验收 / 交接提示，确定性四步细化、后续阶段（状态机/LLM/GM/打包）粗线条占位；编辑器随模拟器逐步同步长面板。

**分库原则**：数据进哪个库看"真人用户会不会有这份数据"。会有（账号/帖/关注/互动/媒体/私信）进 server 独占的 `world.db`、经 API 读写；不会有（模拟器内部机关）单开模拟器独占的 per-world 库、绝不进 `world.db`——NPC 数值与运行时态进 `npc-state.db`，决策轨迹与 GM/Agent 日志进 `sim-trace.db`。决策轨迹由原 JSONL 方案改为 SQLite（better-sqlite3 已在 monorepo、大文件编辑器打不开）。话题这类双重身份实体按字段拆：用户可见部分进 world.db、导演编排元数据归模拟器侧。`m5-npc-state-machine.md` 旧"灌进/活在世界 DB"措辞随之纠正为 `npc-state.db`。

**债务前移**：发现错放/欠债，迁移挪到尽早能做的步而非堆后续。GM/Agent 日志归位前移到 0.3b（建表 + 拆 server 内存态 agentLogs），话题拆分前移到路线图 1.1b。

## 编辑器形态决策

编辑器从 Phase 0 即 Electron 桌面程序（仅编辑器走 Electron，社交站 server 与 simulator 仍各自 dev 进程、由 Electron 连接，整体打包留 M5-6）。多窗格用 dockview。停靠交互经讨论定为 **Adobe 式**（dockview 原生：拖标签到边缘分屏带预览 / 拖到标签条合并 / 拖分隔线调宽 / 浮动 / 弹出独立窗口），不自写 Blender 角落区；每格保留 Blender 式类型下拉。浮动/弹出入口与具体布局打磨待面板补齐后再做。
