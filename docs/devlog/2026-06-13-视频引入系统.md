# 开发日志 2026-06-13 · 外站视频搜索与引入系统

## 概要

补齐媒体系统此前的缺口：视频此前只能本地上传，本日加入从外站搜索与引用视频的能力，与搜图一致地全部经 HTTP API 暴露（供第二阶段虚拟用户使用）。核心是三种引入形态并存、由用户自选，依赖 yt-dlp / ffmpeg 二进制（设置页一键安装）。数据库 schema v12 → v13。分六周期实施，每周期独立提交并经人工验收，其间多次返工修复真实环境问题。

## 三种视频形态

| 形态 | 存储 | 时效 | 适用 |
|---|---|---|---|
| 嵌入式链接卡 | 零（链接卡增强） | 随源 | YouTube/B 站链接，默认形态 |
| 下载入库 | 占世界 media 目录，永久 | 不腐烂 | 想留档的内容 |
| 流式引用（边下边播） | 只存 origin_url + 元数据 + 海报图 | 源失效即不可播 | 大文件、即兴分享 |

形态路由（mode='auto'）：可嵌入站点（YouTube/B 站）默认走嵌入卡不建任务；用户可在设置里对站点逐站覆盖为 download/stream。非可嵌入站点按全局 defaultMode。显式 mode 无条件执行。裁决在服务端（API 优先，虚拟用户同享策略）。

## 各周期详情

### A 嵌入式链接卡

正文贴 YouTube/B 站链接时，链接卡渲染官方 iframe 播放器（X 同款）。`link-cards/embed.ts` 的 `deriveEmbed` 纯函数从 URL 推导 embed 地址：YouTube（watch/youtu.be/shorts/live，`t` 参数转 `start` 秒数，用 youtube-nocookie 隐私域）、B 站（BV/av 号、分 P 透传）；视频 ID 经严格正则校验后才拼接，杜绝 iframe src 注入。`LinkCardView.embedUrl` 由 `viewsForUrls` 现算不落库（推导规则升级后旧帖自动受益，虚拟用户走 API 同样拿到）。前端 LinkCard 门面模式：先显示缩略图 + 播放钮，点击才换入 iframe——滚动时间线对第三方零请求（隐私 + 性能）。

### B 工具二进制管理

`modules/tools`：yt-dlp.exe（GitHub release 单 exe）与 ffmpeg（yt-dlp/FFmpeg-Builds 的 win64 zip，PowerShell Expand-Archive 解压）一键安装/更新，存 `data/bin/`（实例级，不入 git）。下载用全局 fetch（自动跟随 release 302、吃全局代理），停滞 90 秒才判失败（不按总时长杀慢而未死的下载）。安装任务带进度反馈：文件名、来源主机、已下载/总量、1 秒窗口实时速度、旋转动画。下载源可替换为镜像（GitHub CDN 经代理波动极大）。

返工修了两处：① 两工具共用 tmp 目录、任务结束清理整个目录导致并行安装互删 zip → 改每任务独立 `tmp/<id>/`；② Expand-Archive 报错中文路径乱码 → 改 `-EncodedCommand`（UTF-16 base64）+ 命令内切 UTF-8 输出。

### C yt-dlp 封装 + 异步任务 + 下载模式 + 同源去重

`ytdlp.ts`：probe（-J 元数据，含渐进式格式提取）/ searchFlatPlaylist / download（格式选择器 `bv*[height<=H]+ba/b`、`--max-filesize`、进度正则解析）。Windows 下 yt-dlp 派生 ffmpeg 子进程，超时/取消必须 `taskkill /t /f` 杀进程树。`video-tasks.ts`：进程内任务管理器，每用户并发 2 + 排队，终态保留 30 分钟，取消走 AbortSignal；重启丢任务（已入库媒体不丢）。

migration v13：media 表加 `storage`（'library'/'stream'）、`duration_ms`、`poster_media_id` 与 origin_url 索引。海报为独立 image media 行（不挂帖），video 行经 poster_media_id 引用。流式行 file_name 留空 → /file 端点天然 404（零改动防误用）。

下载任务流：probe → 同源去重（同 origin_url 已入库视频复用，本人未挂接直接用、否则硬链接零拷贝新行）→ 下载到 `data/tmp/video/<taskId>/` → 世界热切换核对 → 入库 → 海报非致命补挂。上传视频限额改读配置（multipart 静态上限放宽到 512MB 硬顶，真实限额在 service 校验）。`media-search.json` 加 video 段（maxHeight/maxDownloadMb/defaultMode/hlsFallback/siteModes）。

前端：发帖框 URL 引入加图片|视频切换，任务托盘 IngestTaskTile 占位卡（进度/取消/失败重试），有活跃任务时禁止发布。

### D 流式引用

`stream-resolver.ts`：mediaId → 渐进式 mp4 直链 + 防盗链头的缓存。直链是源站签名的临时地址，过期重解析——这正是"每次播放都新鲜"的机制。`/api/media/:id/stream` 公开代理：解直链 → Range 透传（undici，手动跟重定向逐跳 SSRF 校验，bodyTimeout=0 容忍暂停）→ 回传 200/206 与 content-range/length/accept-ranges，no-store；上游 403/410 失效重解析一次再回 410。路由由 video-search 模块注册（避免 media 反向依赖）。HLS-only 源按 hlsFallback 降转下载或报 HLS_ONLY。

前端：播放器加 poster + onError 流式失效海报占位；视频缩略优先用海报（不触发代理拉流）；Composer/搜索面板加自动/下载/流式模式选择。

D 期返工密集，记录见下文"真实环境问题"。

### E 三搜索源 + 搜索面板视频维度

`adapters/{youtube,pornhub,rule34video}.ts`：YouTube（yt-dlp ytsearchN，免 key 最稳）、Pornhub（搜索页 flat-playlist）、Rule34Video（自抓搜索页 HTML，仿 pinterest 模式，正则非贪婪取到 .time 时长）。`GET /api/video/sources` 与 `GET /api/video/search?q=&source=`（单源直查/全可用源并行 allSettled 静默吞单源失败）。前端 MediaSearchPanel 加图片|视频维度切换，视频结果 16:9 卡（缩略图 + 时长徽标 + 来源角标）+ 引入方式下拉。

**视频源不设内容分级**：与图片源不同，视频源的可见内容由平台性质本身决定（YouTube 天然搜不到 R18、Rule34Video 本身即成人站），分级概念无意义，也不受世界 contentRating 约束。唯一可用性条件是 yt-dlp 是否安装。（初版误按图片思路做成成人源按世界分级封禁，经反馈返工对齐为源恒可用。）

## 真实环境问题与解决

- **B 站 412 风控**：api.bilibili.com 对无 Cookie 请求一律 412。Cookie 必须走 `--cookies`（Netscape jar，域级生效含全部子域）——`--add-header Cookie` 被 yt-dlp 限制为只作用于初始 URL 的域，跨子域 API 带不上。凭证获取做成 CDP 引导登录（同 Pixiv 范式）：点按钮弹浏览器扫码，经 DevTools 轮询 Storage.getCookies，出现 SESSDATA 即捕获全部 bilibili Cookie 自动入配置；手动粘贴 Cookie 降为"高级"兜底。
- **流式直链时效误判**：rule34video/pornhub 的签名直链无 expire 参数却被给 4 小时 TTL，ingest 时 prime 的缓存直链在播放窗口内已失效。改为：有 expire（YouTube）按其计长缓存；无 expire 给 20 秒短 TTL，几乎每次起播都重解析换新鲜直链。
- **流式播放并发 410**：rule34video/pornhub 的签名直链不支持同签名并发，而浏览器播放天然多连接并发拉分段（Composer 单连接预览没事，时间线自动播放多并发才暴露）。StreamResolver 加 per-mediaId 串行锁 runExclusive，controller 把"解直链 + fetchUpstream + 重试"整段串行化（仅获取响应头互斥，body 流回不互斥）。实测 rule34 并发 8 个 Range 全 206（修复前 5/8 失败）。
- **PH 缩略图 403**：phncdn.com 防盗链需 Referer。REFERER_BY_HOST 改为后缀匹配规则 refererForHost（含 pixiv、phncdn），媒体入库与 preview 代理共用；前端视频缩略图 onError 回退 preview 代理。
- **yt-dlp 报错不可读**：失败时完整 stderr + 命令行进服务端 console，任务 errorMessage 取首个 ERROR: 行，任务卡报错放宽三行 + title 全文。

## 已知限制

- **Pornhub 源不稳定**：CDN 风控 + viewkey 短时效，流式播放与部分缩略图仍不可用（与 rule34 的签名直链不同，串行化救不了）。R18 内容用 Rule34Video 已够，PH 暂不修；PH 默认走下载模式（auto 路由），主动选流式才会踩坑。
- 流式播放体验依赖源站：YouTube 流式多为 360p 渐进式（720p 为 DASH 分轨，流式拿不到），高清需下载模式。
- 任务表进程内，重启丢任务（已入库媒体不丢）。
- b23.tv 短链不支持嵌入卡推导。
