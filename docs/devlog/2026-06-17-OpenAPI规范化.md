# 开发日志 2026-06-17 · 子篇：API 以 OpenAPI 规范化

主日志见 `2026-06-17.md`。本篇记录把现有 HTTP API 以 OpenAPI 3.1 规范化、产出完整可协作的 API 文档的工作。采用 **code-first**：spec 从 Fastify 路由 schema 自动生成、与代码同源。覆盖三端：社交站 server、编辑器后端、模拟器（无 HTTP，文字契约）。

## 成果

- 社交站 server（`:3000`）：`GET /docs` 交互式 Swagger UI、`GET /openapi.json`；快照 `docs/openapi/server.{json,yaml}`。**101 路径 / 128 操作**全部含 tag（按域分组）/ 中文 summary / 唯一 operationId / security；66 操作含响应 schema。
- 编辑器后端（`:5176`）：`GET /docs`、`GET /openapi.json`；快照 `docs/openapi/editor.{json,yaml}`，13 条代理/轨迹/布局路由。
- 模拟器：确认无入站 HTTP（API 消费方），改以文字契约表达——指向它调用的 server spec 与它独占写的库 schema（`sim-trace.db`）。说明见 `docs/openapi/README.md`。
- `docs/server-api.md` 顶部加指针指向 `/docs` 与快照；`CLAUDE.md` 运行段加 `gen:openapi` 命令与文档地址。

## 关键实现与踩坑

- **插件加载顺序**：`@fastify/swagger` 的 `onRoute` 钩子只采集其后注册的路由。`buildApp` 改 async、`await registerSwagger(app)` 后再注册路由——否则同步的 `app.get()` 先于 avvio 队列里的插件执行，spec 采到 0 路径。
- **响应组件从 shared 类型生成**：`ts-json-schema-generator` 按显式根类型列表生成 36 个组件（`gen:schemas` → `components.generated.json`），`$ref` 改写为 Fastify `X#`、startup `addSchema`、`refResolver` 用 `$id` 命名。`topRef:true` 才能让非递归根类型本身进 `definitions`（否则被内联在返回顶层漏收）。
- **响应 schema 仅文档不序列化**（核心决策）：Fastify 默认用 `schema.response` 经 fast-json-stringify 序列化并按 schema 过滤字段。递归类型 `PostView.quoted→PostView` 会让序列化器编译时**栈溢出**。解法是 `setSerializerCompiler(() => (d) => JSON.stringify(d))`：响应一律 `JSON.stringify`、忽略 response schema——既规避递归编译，又保证响应负载与挂 response 前**完全一致**（实测全站流 PostView 全字段无丢）。请求体/参数校验仍走 Ajv。为稳妥，组件 schema 一律 `additionalProperties:true`。
- **编辑器后端拆 app.ts/index.ts**：`buildEditorApp`（纯组装、无副作用）移入 `app.ts`，`index.ts` 仅监听，使 gen 脚本可静态 import 而不触发监听。editor 无 `"type":"module"`，tsx 按 CJS 转译 → gen 脚本不能用顶层 await，包 async IIFE。
- **批量改 16 模块路由元信息 + 响应**：用并行子代理按模块分工（每模块独立文件无冲突），主代理建样板 + 校验（128 操作零 operationId 冲突、无鉴权项恰为公开/worlds/simulator 集合）。

## 维护方式

改路由后跑 `npm run gen:openapi -w @socialsim/server`（先刷响应组件再出快照）/ `-w @socialsim/editor`，提交快照使仓库 spec 与代码一致。响应形态真相源仍是 `shared/src/types/`。
