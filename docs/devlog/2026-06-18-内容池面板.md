# 开发日志 2026-06-18 · 内容池面板（Phase 1.6 + 界面打磨）

Phase 1.6「内容池面板」的实现与按用户反馈的多轮界面打磨。主日志 `2026-06-18.md` 留概要，细节在此。

## 架构决策：编辑器↔模拟器控制通道

1.6 第一个编辑器侧面板。预览要"所见即所得"——预览结果须与模拟器真发帖一致。组装引擎在模拟器里，编辑器按设计不 import 模拟器。曾考虑把引擎抽到公共包共用，被用户依"为必然到来的需求设计"否决：内容生成必然演进到依赖 NPC 活状态 / LLM，忠实预览须用模拟器的活引擎与活状态算，故引擎**留在模拟器**、编辑器**要结果**。

定为：模拟器开一个仅绑 127.0.0.1 的小控制接口（端口入基础设施配置，默认 5177），编辑器后端代理预览请求过去；二者隔 HTTP 边界、互不 import，与三条数据线同源。模拟器未运行时预览给优雅提示。落档 `docs/m5-x-re-plan.md`「编辑器↔模拟器控制通道」。

## 实现

- **模拟器**：内容池存为 provider 可热替换；新增 `control/control-server.ts`（node:http，`GET /health` + `POST /preview` 用活引擎组装草稿池）；fs watch 监视世界与全局池子目录（防抖 300ms）热重载，编辑器存盘即生效、不重启。`config.ts` 加 `controlPort`。
- **编辑器后端**：`pool-files.ts` 读/写世界与全局层三层池文件；`app.ts` 加 `GET /api/content-pools`、`POST /save`、`POST /delete`、`POST /preview`（代理到模拟器控制端点，sim 未运行返 503）。
- **编辑器面板**：`ContentPoolPanel.tsx` 浏览 + 增删改三层 + 预览。

运行时冒烟通过：预览经活引擎产出原神/鸣潮/水贴帖；读三层 provenance 正确；save 写文件 + 模拟器热重载触发 + delete 干净。期间发现并记下两个测试坑：①curl.exe 在 Windows 传命令行中文参数会乱码（用 `--data @文件` 避开）；②环境里堆了大量重复/孤儿 node 进程致端口占用（5177 被游离编辑器后端占）。

## 按用户反馈的多轮打磨

- **一条一文件**：原把多个组件/语法塞一个文件（图省事）被否。改为每个组件/语法/池各一文件、文件名即名字；编辑器保存按 名字 + 层 + 全局/世界 推导落点，作者无需关心文件归属。把现有 demo 与全局层打包文件（base/原神/鸣潮.json）拆成一条一文件。
- **提示词去题材化**：面板 placeholder 不再出现具体作品名，改中性。
- **账号类型押后**：池"准用门槛"仍用 1.5 的临时 `tier`（core/ambient），未切到五预设驱动模式——按用户意见押后到 1.7 连驱动模式字段一起改，记入 roadmap 1.7 与 `m5-account-model` 落地点。
- **语法卡片条**：语法编辑从竖列下拉行重做为横向卡片条（左→右=成文顺序，拖卡片调序）。槽位支持：从组件区**拖组件入槽**、**多选一**（槽内随机出其一，可设**每组件权重**）、**出现概率**滑块、**互斥组**。组件选择改小选择窗（按分组、可搜索、勾选）。互斥不手填组名、改**勾选其它槽位**（底层自动编组）。
- **互斥语义改顺序优先**：原"全员掷后在通过者里随机挑"分布不对。改为**组内按槽位顺序、首个掷中者胜出**：靠前的先掷其出现概率，中了则靠后的因互斥不出；都没中则该组不出。例 夸赞70%+贬低100% → 70% 夸赞 / 30% 贬低；贬低改 50% 则 15% 两者都不出（冒烟实测吻合）。互斥成员仍各自配概率。
- **高级概率表达式撤掉**：表达式（如 `slangDensity`）引用的 NPC 性格变量未接通、暂等同固定 50%，按"只摆做了的"先从 UI 撤掉；数据层仍支持、加载不丢。表达式功能（固定值 vs 表达式、待状态机接变量）与互斥顺序语义落档 `re-plan`「槽位出现控制」。
- **"出现概率"改名"槽位出现概率"**：与组件权重区分（前者管槽出不出，后者管出哪个组件）。

## 布局误解与校正

用户说的"内容池默认 layout（左内容池 | 右上内容池 | 右下预览器）"指 **dockview 工作区布局**，被误解为在面板内部硬塞三区。校正：

- 内容池**复原为单一面板**（一个浏览器：左 tab + 分组列表 | 右 编辑器），删内部三区。
- 预览器抽成**独立面板** `PreviewerPanel`；预览结果经 `preview-bus` 跨面板推送。
- 「内容池」**预设布局**（`layouts/presets`）编排：左 内容池(语法) | 右 内容池(组件，`params.tab`) / 右下 预览器。

## 预览器可视化

- 随机模拟一条，按槽位拆块：出现的槽=彩色虚线框 + 文本，下方连线标所选组件名；**未出现的槽=红色虚线框 + 候选组件(划掉) + 原因（被互斥 / 概率落选）**。顶部显示本次用的语法（池子预览即可看出随机选了哪套语法）。「再来一个」重掷。
- 为此组装结果加 `segments`（覆盖**全部**槽位，含未出现的及原因）；`resolveGroups` 返回 `{ winner, failed }`（掷过没中=概率落选；未掷的非 winner=被互斥）。shared 加 `PreviewSegment`。

## 跨面板共享态（两处同类修复）

用户两次指出跨面板数据不同步，确立原则：**同一份数据所有面板共用一份活的来源，不各存快照**。

- **内容池数据**：左语法面板与右上组件面板原各存一份，改了分组后另一面板的"选组件"弹窗不刷新。提到模块级共享存储 `pools-bus`，所有内容池面板订阅同一份，任一处保存即全局刷新。
- **预览来源**：「再来一个」原重放上次"预览"时的草稿快照，改了语法没再点预览则重掷仍用旧设置。改为预览总线存"当前预览来源"——一个每次渲染更新、实时读编辑器最新草稿的函数（编辑器经 ref 注册）；重掷实时取最新草稿再组装。

这两处与 `selection.ts`（时间轴选中态）同范式：编辑器跨 dockview 面板共享态一律走模块级 pub/sub 单例，不复制快照。
