# 模拟器真实使用契约

模拟器是网站 HTTP API 的通用消费者。本契约规定它必须满足的真实使用标准，以及判定"是否达标"的金标准端到端场景。建模基底层（NPC 内态决策）的设计依据是 `m5-npc-state-machine.md`；本契约只治理基底之上的集成、运行与验收，二者不重叠、不矛盾。

## 通用消费者原则

- 模拟器不得修改网站（server/client）代码，只能通过公开 HTTP API 操作世界。
- 模拟器必须用尽网站已有的 API 能力：发帖、回复、引用、媒体配图/视频、关注、点赞、转发、通知轮询。任何"真人能在网站上做、虚拟用户却做不到"的能力缺口都算未达标。
- 虚拟用户与真人走完全相同的 API 路径，模拟器不享有绕过业务规则的内部通道。

## 跟随活动世界

- 服务端在任一时刻只有一个活动世界。模拟器必须驱动当前活动的那个世界，语义与网页端世界切换一致。
- 服务端切换活动世界时，模拟器必须自动完成：把旧世界的运行时状态强制落盘（flush）→ 重新登录新世界的账号 → 重载新世界的全套配置 → 在新世界继续运转。整个过程不重启进程。
- 模拟器不得带着旧世界的配置或令牌对新世界空转，不得向已切走的旧世界连接写入任何数据。
- 模拟器不得把 `worldId` 烧死在启动参数里。启动时查询当前活动世界，运行期跟随活动世界变化。

## 世界自包含的配置加载

- 一个世界的全部驱动配置必须从该世界自己的文件夹加载：tuning override、faction 注册表、剧本、三类内容池、npc-profiles、账号 roster。
- 复制一个世界文件夹必须得到一个完整可被驱动的世界，无需额外的进程级启动配置。
- 模拟器的启动参数只允许包含与世界无关的基础设施配置（API 地址、admin key、RNG 种子等），不得包含任何特定世界的内容数据。

  具体反例：当前把内容池写在 simulator 启动 config 的 `contentPool` 字段、把账号 handle/password 写在启动 config 的 `accounts` 字段，均违反本条——这些数据必须随世界文件夹走。

## 账号模型

- 网站侧所有账号结构一致，不按"是否虚拟"做身份区分。虚拟身份只用 `users.is_bot` 运营标记表达。
- 账号的 `handle` / `displayName` / `id` 必须是拟真名称，不得编码 bot 身份（不得出现 `sim_`、`_amb_`、`bot`、序号后缀、A/B/C 字母后缀等暴露虚拟身份的命名约定）。
- 创建虚拟账号走管理端代理建号接口，建号时设 `is_bot=1`。公开注册接口不承担批量建号职责。
- 盲评原则：喂给 LLM 的任何内容（世界摘要、候选帖、收到的 @ 与回复）必须剥除 `is_bot` 及一切可区分真人与 NPC 的标记。真人身份只在引擎层使用，不进提示词。

## 发帖形态忠实

- 一条内容的形态（顶层帖 / 回复 / 引用）必须与 NPC 当前行动一致。标注为回复的内容不得当顶层帖发出，标注为顶层帖的内容不得当回复发出。
- 候选内容过滤必须包含 shape 维度：顶层发帖只取 `standalone`，回复动作只取 `reply`，引用动作只取 `quote`。
- Beefing 状态发帖优先 quote 对手，InThread 状态发帖优先 reply 串中现有成员。
- 内容 `形态`（standalone/reply/quote，池维度）与剧本 `scriptedDraft.shape`（post/reply/quote）的映射必须明确且一致：standalone 对应顶层帖；interjection 形态的片段只作占位符料、不单独成帖。

## 媒体能力

- 自动发帖必须能像真人一样携带图片/视频，链路为：搜索媒体 → 入库（`POST /api/media/from-url`）→ 携带 mediaIds 发帖。规则驱动路径与 LLM Agent 路径都必须能配图。
- 媒体获取失败时必须降级为纯文本帖，不得阻断发帖。
- 配图概率、单帖媒体数上限、媒体源偏好必须是可调常量，从 TuningService 取。
- 媒体内容分级由世界 `contentRating` 经服务端 media-search 把门，模拟器不重复实现分级判断。

## 内容池三类布局

- 内容池按三类分文件存放并按活动世界加载：
  - 基础原子池（全局共享，入 git）：`data/global-pools/<name>.json`
  - 场景特化池（世界级）：`data/worlds/<id>/scene-pools/<name>.json`
  - 话题专属池（临时）：`data/worlds/<id>/topic-pools/<topicId>.json`
- 场景池与话题池必须从当前活动世界的文件夹加载，随世界切换换源。

## 可观测与可恢复

- 运行时状态（mood/memory/activity/attention/relationship）按世界分目录持久化，写盘无 BOM 且原子化。
- 进程重启或崩溃后，靠启动时 gap mitigation 恢复连贯，不靠叙事合理化。
- 每一步对世界的写操作可记录、可审计。

## 金标准端到端验收场景

达标判定不看纯函数断言数量，而看下面这个场景能否在一个**全新创建的世界**（非 modern-earth、非 acgn 预置世界）上跑通。每一步都是可观测断言。

1. 经管理端 API 从空白创建/初始化一个新世界 W1，并激活它。
2. 经管理端代理建号接口创建若干账号，全部 `is_bot=1`，handle/displayName 为拟真名称（断言：数据库 `is_bot=1`，且名称不含 bot 命名约定）。
3. 为 W1 写入该世界自包含的配置：tuning override、factions、组件类型库 / 语法库、含 intent/shape 维度的场景池、npc-profiles、账号 roster。
4. 启动模拟器（启动参数不含任何 W1 特定内容数据）。断言：模拟器查到 W1 为活动世界，登录 W1 账号并开始运转。
5. 运转一段模拟时间后断言：时间线上既有顶层帖又有回复，且回复确实挂在被回复帖下（帖/回复形态忠实）；至少一条帖按场景携带了入库媒体（`GET /api/media/:id/file` 返回非空字节）；不同 alignment 的 NPC 发帖语气可区分。
6. 服务端激活另一个世界 W2。断言：模拟器自动 flush W1 状态、停止驱动 W1、重登 W2 账号、用 W2 的配置发帖；全程无 401 空转，无对 W1 的写入。
7. 服务端激活回 W1。断言：模拟器恢复驱动 W1，NPC 运行时状态从落盘点续上（gap mitigation 生效）。

## 验收方法论

- 头牌验收基线是上述端到端契约场景对全新世界跑通，而非纯函数断言条数。纯函数与 schema 验证脚本降级为单元层，不再作为"达标"的主证据。
- 每一个真实使用面（跟随活动世界、媒体、帖/回复形态、账号模型）必须有契约级断言覆盖，不止单元级。
- 验收脚本不得依赖任何预置世界。需要世界时经 API 现建，跑完清理。
