# M5-X 重启推进方案

## 背景

M5-X 首轮实施（NPC 状态机方向）产生了接近推翻重来的理解偏差，已整体回退至 M5-4 完成态。当前分支 `feat-M5-X-RE` 在此基础上重启。

回退暴露的核心问题：首轮以完整 NPC 状态机为目标直接铺开，而唯一产出物是网站上一条与普通帖子无异的帖子。模拟器选了哪个池、是否正确区分帖与回复、状态机是否生效、媒体逻辑是否运行，全程不可在网站或编辑器中识别。每个阶段对用户都是黑箱。

## 核心原则

本轮全程遵守以下原则，任一被违反即视为偏离方向：

- **先确定性后 LLM**：tick 层必须做成零 LLM 纯规则——行为是（人设 + 池子 + 状态 + tuning + RNG 种子）的纯函数，给定种子可复现、可解释。LLM Agent 路径先从关键路径移除，待确定性层验收认可后再叠加。前期 NPC 内容朴素（模板 / 池子）是可接受的代价。
- **编辑器为唯一观察窗，与模拟器同里程碑同步交付**：每推进一段模拟器逻辑，必须同步给编辑器长出能看见这段逻辑的面板。不允许出现"模拟器已实现但用户无法在编辑器中观察"的功能。
- **每里程碑配可观测验收**：达标判定不看纯函数断言条数，而看"打开编辑器或网站，能看到并一眼判断对不对"。纯函数与 schema 验证降级为单元层。
- **可调值不硬编码**：衰减率 / 系数 / 阈值 / 概率从第一行起抽到 tuning 配置（全局 defaults + 世界级 override），不允许写死在 `.ts` 中。
- **账号模型统一**：网站侧所有账号结构一致，虚拟身份只用 `users.is_bot` 运营标记表达，不影响任何业务逻辑或 API 行为。
- **盲评**：喂给 LLM 的任何内容必须剥除 `is_bot` 及一切可区分真人与 NPC 的标记。真人身份只在引擎层使用，不进提示词。

## 架构锁定

### 三条数据线

| 数据线 | 通道 | 说明 |
|---|---|---|
| 改世界 | 模拟器 → server 公开 / admin HTTP API | 模拟器只通过此通道写世界，与真人走相同 API，不享有内部捷径 |
| 观测世界 | 模拟器 → 编辑器后端 → renderer | 决策轨迹等观测数据走旁路，不经社交站 server |
| 配置随世界 | 世界文件夹 `data/worlds/<id>/` | 驱动配置随世界文件夹走，复制文件夹即得完整可驱动的世界 |

三线分离使社交站 server 不沾任何模拟器观测概念，模拟器不持有任何特定世界的启动数据。

### 编辑器工程形态

编辑器为工作室 / IDE 多窗格形态，可同屏并列多个面板（如时间轴 + 决策轨迹 + 检视器）。工作区从重建第一刻起即为 renderer + server 两层：

```
editor/
├── src/
│   ├── server/    # 编辑器后端：renderer 的唯一数据源，聚合/代理 server admin API，承载决策轨迹接入与 SSE 推流
│   └── renderer/  # 多窗格前端
└── （main/ Electron 主进程留至 M5-6）
```

编辑器后端是 renderer 的唯一数据源，后续的时间轴聚合、Agent 会话管理均长在此层。renderer 不直接访问数据库，不绕过编辑器后端调社交站 server。

### 决策轨迹

模拟器每次写世界（发帖 / 回复 / 引用 / 赞 / 转 / 关注）必须吐一条结构化轨迹，记录"为什么这么做"。轨迹按世界落盘为追加式 JSONL（`data/worlds/<id>/sim-trace.jsonl`，无 BOM）作为可审计的持久真相源；编辑器后端按时间区间索引它供时间轴渲染，新事件经 SSE 实时推送。

轨迹不另做独立滚动日志——每条轨迹即时间轴面板上的一个块：纵轴为账号轨道、横轴为模拟时间，块落在该账号该 `simTime` 的位置，点开块展示该条的"为什么"。"决策轨迹视图"与"时间轴面板"是同一面板。

轨迹事件至少包含以下字段：

```jsonc
{
  "at": 1718500000000,        // 真实时间
  "simTime": 1718500000000,   // 世界模拟时间
  "entity": "alice",          // 驱动账号 handle
  "action": "post",           // post | reply | quote | like | repost | follow
  "activityState": "Browsing",
  "intent": "earnest",
  "shape": "standalone",
  "poolId": "coser-praise",
  "entryId": "p_0042",
  "mediaAttached": true,
  "mediaReason": "scene=coser 命中配图概率",
  "targetPostId": null        // reply / quote 时为被作用帖 id
}
```

轨迹 sink 地址属于与世界无关的基础设施配置，允许进模拟器启动参数。sink 缺失时模拟器仍可独立运行，仅本地落盘不推流。

## 内容池模型

内容池不存完整整句，存可组装的片段，组织方式与模拟器 ECS 同构。组装权重、槽位概率等系数从 tuning 取，不写死。

### 三层结构

| 层 | ECS 对应 | 定义 |
|---|---|---|
| 组件类型 | Component | 作者自由定义的可复用槽位种类，自带候选片段库。例：`开场`=["话说","刚刷到"]、`黑话收尾`=["确实","笑死"] |
| 语法 | Archetype（实体组装式） | 有序引用一批组件类型 = 一种句式骨架；每个引用可标可选或按概率出现 |
| 池 | spawn 规格 | 由维度定义，声明用哪几套语法及其权重 |

组件类型库与语法库均为可跨池复用的注册表。生成一条帖子 = 按池选一套语法（ECS 的实体），逐组件各取一个片段填充（组件实例值），即得一条具体内容。

**槽位平等**：语法中每个槽都是平级组件，无 opener / body / tail 之类内建特权角色或固定层级；顺序只决定文本左右位置。语法引用具体组件类型还是更通用的组件类型，纯由作者按需要的颗粒度决定，不是引擎层的不同机制——`原神主体` 与通用 `主体` 都只是组件类型，区别仅在作者切分的粗细。

### 维度

每个池由一组维度标签定义。`形态` 为全局固定维度（standalone / reply / quote，对应顶层帖 / 回复 / 引用）；`领域` / `作品` / `模式` 等其余维度由各世界自己的标签注册表定义，与 faction 注册表同范式（修真世界用门派 / 功法，不用游戏 / 原神）。`模式` 即粗粒度 intent（正常讨论 / 玩梗 / 抬杠 / 阴阳）。

### 示例

```jsonc
// 组件类型库（可跨池复用，自带候选片段）
{
  "开场":   ["话说", "刚刷到", "不是，"],
  "原神主体": ["雷神", "新版本", "本期深渊"],
  "正面评价": ["强度真不错", "建模吹爆", "剧情有点东西"],
  "黑话收尾": ["确实", "笑死", "蚌埠住了"]
}
// 语法库（引用组件类型，可跨池复用）
{
  "感叹式": { "slots": [
    { "component": "开场",   "optional": true },
    { "component": "原神主体" },
    { "component": "正面评价" },
    { "component": "黑话收尾", "prob": "slangDensity" }
  ] },
  "直球式": { "slots": [ { "component": "原神主体" }, { "component": "正面评价" } ] }
}
// 池（维度 + 语法引用）
{
  "dimensions": { "作品": "原神", "模式": "正常讨论", "形态": "standalone" },
  "grammars": [ { "ref": "感叹式", "weight": 2 }, { "ref": "直球式", "weight": 1 } ]
}
```

组装：感叹式 → "刚刷到本期深渊建模吹爆，笑死"；直球式 → "雷神强度真不错"。约二十个组件片段经多套语法可组装出数百条不重样内容。

### 占位符即内联组件引用

片段文本内的 `{key}` / `{key:variant}` 占位符（如 `{黑话}`、`{target_faction:贬}`）就是在该位置内联取一个组件的值，与槽位是同一机制的两种位置——槽位为位置式组件引用，占位符为内联式组件引用。占位符解析不到匹配项时丢弃该候选片段重抽，不输出病句。

### 三类池布局

内容池按三类分文件存放并按活动世界加载：基础原子池（全局共享、入 git）`data/global-pools/<name>.json`；场景特化池（世界级）`data/worlds/<id>/scene-pools/<name>.json`；话题专属池（临时）`data/worlds/<id>/topic-pools/<topicId>.json`。组件类型库与语法库同样按"全局共享 / 世界级"两层存放，全局层入 git。

### 篇幅维度

`形态`（standalone / reply / quote）之外，池另设独立的**篇幅**维度，因为长短帖生成方式根本不同：

- **短帖**（一两句）：词 / 短语碎片拼装，少量碎片拼出海量不重样，重复可容忍。
- **松散长帖**（清单体 / 几点想法 / 测评 / 速报）：把同一套 ECS 模型抬到**段落粒度**——组件是段落块、语法是文章结构（如 测评 = [开场感受] + [优点段] + [缺点段] + [总结段]，每槽从对应段落池抽）。仍零 LLM，各段相对独立故连贯性够用。
- **紧密长帖**（有论证主线 / 叙事 / 来龙去脉）：确定性拼不出，归 LLM 单独调用，或为签名角色预写整篇模板。

长帖重复比短帖刺眼，段落池须备足量；长帖本就低频，中等量即可。某 NPC 的签名长帖格式（媒体号速报体 / 测评人测评体）可做成其私有长帖语法，使长文也一眼认得出作者。

### NPC 私有池

共享池（全局 / 场景 / 话题）之上，单个 NPC 可有**私有池**，优先级最高，让 NPC 不靠 LLM 也精确成"它自己"而非泛泛账号：

- **私有说话池**：只属于该 NPC 的组件 / 语法 / 片段（口头禅、签名句式），存其 NPC 文件夹的 `pool/`。一个旋钮定独占度——**独占**只用私有池（最像具体角色）/ **优先**先私有再退回共享（省事）；默认优先。
- **优先媒体池**：从世界媒体库挑一组该 NPC 专属的图存其 `media/`，发图帖优先取，使其图帖风格一致（如某 coser 永远发它那套照片）。

私有池填得越满，NPC 越精确且全程零 LLM——重要角色填厚、路人号留空吃共享池。关系阶段的"指名子池"偏置可指向私有池，实现对特定人用专属话术。

## 里程碑阶梯

状态机（mood / FSM / alignment / 撕逼蔓延）后置。先用以下四步把确定性地基与可观测闭环打通，每步代码与编辑器面板同步交付：

| 步 | 模拟器侧 | 编辑器同步长出 | 可观测验收 |
|---|---|---|---|
| **0 地基** | 配置随世界走 + 跟随热切换 + 代理建号 + 决策轨迹基建 | 多窗格壳层 + 编辑器后端骨架 + 控制台面板 + 最小时间轴（账号轨道 + 事件块） | 网页端切世界后模拟器自动 flush / 重登 / 换配置，时间轴随之换流；DB 中模拟器驱动账号全 `is_bot=1` 且名称拟真 |
| **1 顶层帖** | NPC 按人设从指定池发 standalone 帖；氛围号发水贴；shape 过滤先只放行 standalone | NPC 设计器 + 内容池面板（维度定义 + 多槽位语法 + 模块集编辑） | 时间线全是顶层帖无错位回复；每个时间轴块可追溯到账号 · 池 · 语法 · 模块 |
| **2 配图** | 发帖自然引入 / 不引入媒体，引入时匹配场景；获取失败降级纯文本 | 决策轨迹标注配图原因；按需补媒体库面板雏形 | 时间线部分帖图文相称，`GET /api/media/:id/file` 返回非空字节 |
| **3 回复** | NPC 回复前帖、回复楼中楼，shape 正确，内容贴合被回复帖 | 对话串 / 轨迹视图可见回复挂对父帖 | 对话串自然生长且挂对父帖；轨迹标注回应哪条 · 用了哪个 reply intent |

四步稳定后，状态机按 M5-X.0（tuning 层）→ .1（mood + memory）→ .2（relationship + activity FSM）→ .3（戏剧态 + 撕逼蔓延）→ .4（剧本协作）逐层叠加，每层验收标准为"世界明显更不机械，且变化点可被指认"。

## 形态忠实约束

内容形态（顶层帖 / 回复 / 引用）必须与 NPC 当前行动一致，候选内容过滤必须含 shape 维度：

- 顶层发帖只取 `shape=standalone`，回复动作只取 `shape=reply`，引用动作只取 `shape=quote`。
- `interjection` 只作占位符料，不单独成帖。
- 标注为回复的内容不得当顶层帖发出，标注为顶层帖的内容不得当回复发出。
- Beefing 状态发帖优先 quote 对手，InThread 状态发帖优先 reply 串中现有成员（状态机阶段生效）。

## 账号模型约束

- 创建虚拟账号必须走管理端代理建号接口，建号时设 `is_bot=1`。公开注册接口不承担批量建号职责。
- `handle` / `displayName` / `id` 必须为拟真名称，不得编码 bot 身份。禁止 `sim_`、`_amb_`、`bot`、纯序号后缀、A / B / C 字母后缀等暴露虚拟身份的命名约定。
- 账号 roster 随世界文件夹走，不得写在模拟器启动参数中。

## 配置自包含约束

- 一个世界的全部驱动配置（tuning override、faction 注册表、剧本、三类内容池、npc-profiles、账号 roster）必须从该世界自己的文件夹加载。
- 模拟器启动参数只允许包含与世界无关的基础设施配置（API 地址、admin key、RNG 种子、轨迹 sink 地址），不得包含任何特定世界的内容数据。
- 模拟器不得把 `worldId` 烧死在启动参数里。启动时查询当前活动世界，运行期跟随活动世界变化。
- 服务端切换活动世界时，模拟器必须自动 flush 旧世界运行时状态 → 重登新世界账号 → 重载新世界配置 → 在新世界继续运转，全程不重启进程、不向旧世界连接写入数据、不带旧令牌对新世界空转。

## Step 0 实施范围

### 模拟器侧

- 移除启动 config 中的 `worldId` / `accounts` / `contentPool` 字段，改为运行时查询活动世界并从世界文件夹读取 roster 与配置。
- 实现活动世界跟随：检测活动世界变化 → flush → 重登 → 重载 → 续转。
- 修正时间基错误：当前 `scheduleNext` 以 `Date.now()` 设 `nextActionAt`，又与 `ctx.simTime` 比较，世界非 1 倍速时二者发散；统一为模拟时间。
- 定义决策轨迹事件 schema，每次写世界吐一条，落盘世界文件夹并推送轨迹 sink。

### server 侧

- 确认 / 补全代理建号接口：设 `is_bot=1`、拒绝 bot 命名约定、公开注册不承担批量建号。

### 编辑器侧

- 重建工作区为 renderer + server 两层，建立多窗格壳层与编辑器后端骨架。
- 控制台面板：活动世界 / 时钟 / 模拟器启停 / 切世界态。
- 最小时间轴：纵轴账号轨道、横轴模拟时间，模拟器轨迹经编辑器后端 SSE 实时长为事件块，点开块展示该条决策详情。

### Step 0 验收

将活动世界从 W1 切到 W2 后断言：编辑器控制台显示模拟器自动 flush W1、重登 W2、换 W2 配置，决策轨迹视图随之换流，全程无 401 空转、无对 W1 的写入；管理端查询 DB，模拟器驱动账号全部 `is_bot=1` 且 handle / displayName 不含 bot 命名约定。

## 验收方法论

- 金标准验收基线为上述各步可观测断言在一个全新创建的世界（非 modern-earth、非 acgn 预置世界）上跑通，而非纯函数断言条数。
- 每一个真实使用面（跟随活动世界、媒体、帖 / 回复形态、账号模型）必须有契约级断言覆盖，不止单元级。
- 验收脚本不得依赖任何预置世界。需要世界时经 API 现建，跑完清理。
