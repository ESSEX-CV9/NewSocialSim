# 账号驱动模型

定义一个账号"由谁驱动、产出从哪来、能否被 LLM 接手"的统一模型，作为建号流程与 NPC 设计器的字段依据。

## 统一原则

- **账号类型即默认驱动模式，是运行时模式而非固定身份。** 同一账号在运行中可被 GM 临时抬档、被资源不足下压，类型字段只存"默认基线"，不锁死当下由谁驱动。
- **真人账号是唯一铁墙：GM 与模拟器永不触碰真人账号。** 其余所有账号 GM 均可视情况介入。
- **状态机是贯穿所有非真人账号的底座。** mood / 活动 / 关系 / 记忆等数值（零 LLM）对纯 LLM、半 LLM、纯模拟器账号一律照常维护；驱动模式只决定"谁消费这些数值来产出动作、内容从哪来"，不决定要不要维护它们。一次性氛围号不维护这些数值。
- **实际驱动者按动作运行时解析**：`effective = resolve(默认模式, GM 升级, 资源状态, 真人在场, 降级策略)`。默认模式是基线，GM 往上抬、资源不足往下压或令其闭嘴，真人账号不参与解析。

## 四条轴

每个账号由以下四条正交的轴定义。"账号类型"是这四条轴的几个常用组合，作者建号时选一个命名预设，引擎内部展开成轴。

| 轴 | 取值 | 含义 |
|---|---|---|
| 归属 | 真人 / 被驱动 | 真人 = 有真人登录操作，铁墙；被驱动 = 无真人、由模拟器/GM 驱动 |
| 内容来源 | 真人 / LLM 单调 / LLM 批量 / 池组装 / 无 | 该发帖或回复时，正文由谁生成 |
| 互动来源 | 真人 / 引擎 / LLM | 赞 / 转 / 关注由谁决定 |
| 内态 | 持久 / 无 | 是否维护 mood / 记忆 / 关系（决定有无 `npc-state.db` 行） |

另有一条与四轴正交的**降级地板**轴：

| 降级策略 | 含义 |
|---|---|
| `degrade`（默认） | 资源不足时退到更低档（LLM → 池组装），照常产出，世界变平庸但不停转 |
| `silent` | 资源不足时跳过本次动作、不产出，宁可这条内容不出现，也不出 OOC 内容 |

## 命名预设

| 预设 | 内容来源 | 互动来源 | 内态 | 常配降级策略 | 用途 |
|---|---|---|---|---|---|
| 真人账号 | 真人 | 真人 | — | — | 用户自己的账号 |
| 纯 LLM | LLM 单调 | LLM | 持久 | `silent` | 极重要核心 NPC，用户在意其任何 OOC（含点错赞） |
| 半 LLM | LLM 单调 | 引擎 | 持久 | `degrade` | 内容须符合人设、互动可交引擎的核心 NPC |
| 纯模拟器 | 池组装 | 引擎 | 持久 | `degrade` | 状态机驱动的常规账号 |
| 一次性氛围 | 池组装 / 批量 | 极少 / 无 | 无 | — | 发一条顶层帖填氛围 / 在某串下发一条评论凑数 |

`degrade` 与 `silent` 不绑死预设：半 LLM 账号也可设 `silent`（宁可不发也不发模板帖），但 `silent` 最常用在纯 LLM 核心 NPC 上。

## 状态机的消费者随模式而变

四种非真人账号都跑同一套状态机维护数值，区别在谁读这些数值产出动作：

- **纯模拟器**：引擎读数值，直接产出内容（池组装）与互动。
- **半 LLM**：引擎读数值产出互动；该发帖 / 回复时把内容生成交给 LLM，LLM 也读得到这些数值作为上下文。
- **纯 LLM**：引擎只维护数值、不替它行动；内容与互动都由 LLM 读着这些数值产出。
- **一次性氛围**：不维护数值，状态机基本不参与。

## 纯 LLM 退出确定性互动层

- 引擎绝不替纯 LLM 账号自动点赞 / 转发 / 关注。纯 LLM 的全部互动只来自 LLM 决定。
- 互动 / 级联系统必须维护"自动操作资格"——半 LLM、纯模拟器、一次性氛围号在内，纯 LLM 在外。级联给候选账号派延迟反应判定时，须先排除无自动操作资格者。
- 这是纯 LLM 为高保真付出的代价：用户对其形象敏感到点错一个赞都算 OOC，故它的互动不能交给不读人设的引擎。

## 运行时驱动解析

按动作解析当前由谁驱动，规则如下：

- **真人账号**：始终真人，不进解析，GM 与模拟器不产出任何代其行动的请求。
- **GM 升级（往上抬）**：GM 全权决定、跨账号、时限性的覆盖，临时把某被驱动账号抬到 LLM 档处理一两个动作，完事落回基线。升级不写进账号字段。触发沿用 GM 唤醒机制——真人发帖 / @ NPC / 私信 NPC 时 GM 被唤醒，可对被 @ 的纯模拟器账号临时接手给出符合人设的真回复。
- **资源降级（往下压）**：LLM 预算或速率不足时，按账号降级策略处理——`degrade` 退到池组装继续产出，`silent` 跳过本次动作。日程类发帖软过期后直接丢弃，不补发。
- **优先级**：对真人的响应（@ / 私信）为最高优先级，故核心 NPC 回真人几乎不会因资源饿死；`silent` 是极端情况下的硬保证。

## 分库映射

判据沿用分库原则——真人用户会不会有这份数据：

| 账号 | `world.db` | `npc-state.db` | `sim-trace.db` | `rp.json` / `lore.md` |
|---|---|---|---|---|
| 真人账号 | ✅ | — | — | — |
| 纯 LLM | ✅ | ✅ | ✅ | ✅ |
| 半 LLM | ✅ | ✅ | ✅ | ✅ |
| 纯模拟器 | ✅ | ✅ | ✅ | — |
| 一次性氛围 | ✅ | — | ✅ | — |

- **有无 `npc-state.db` 行等价于有无内心活动**：持久内态账号有，一次性氛围号无。
- **`rp.json` / `lore.md`（RP 资料）仅 LLM 档账号需要**，池组装账号不读它，故纯模拟器与一次性氛围号无此文件。
- 一次性氛围号的发帖动作仍写一条 `sim-trace.db` 决策轨迹，但不建 `npc-state.db` 行。

## 落地点

- 归属 / 默认模式 / 降级策略 / factions / poolAffinities 为 npc-profile 的核心字段。
- 字段在管理端代理建号流程与编辑器 NPC 设计器中设置（路线图 1.7）。
- 公开 API 响应不含任何驱动模式字段，与不含 `isBot` 同理——前端对所有账号一视同仁，不据驱动模式区分或标记。
